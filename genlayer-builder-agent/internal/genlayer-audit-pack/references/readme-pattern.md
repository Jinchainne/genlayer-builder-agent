# README Pattern

Use this structure when rewriting a GenLayer builder README.

## Recommended Order

1. project name and one-line positioning
2. live links
3. why this belongs on GenLayer
4. what the contract does
5. what the user flow is
6. what reviewers can verify quickly
7. setup and verification commands
8. tests and CI

## Mandatory README Claims

The README should make these easy to see:

- this is a GenLayer-native adjudication or judgment workflow
- the contract uses meaningful non-deterministic execution
- the app has a real client path
- the accepted result is written back on-chain
- reviewers can run checks locally

## Tone

Preferred:

- specific
- technical
- builder-friendly
- reviewer-aware

Avoid:

- hype
- vague promises
- empty “AI agent” wording
- claims that are not provable from the repo
